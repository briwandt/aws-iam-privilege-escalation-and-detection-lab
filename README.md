# AWS IAM Privilege Escalation and Detection Lab

## Overview

This lab demonstrates how an AWS identity can gain **indirect access** to protected S3 data through an AWS Lambda function, even when the identity does not have direct `s3:GetObject` permission.

The lab focuses on the security risk created when an identity can:

- Invoke a Lambda function
- Create or modify Lambda functions
- Pass an IAM execution role to Lambda
- Use a service role with permissions the identity does not directly possess

## Lab outcome

The restricted developer identity was unable to retrieve the protected S3 object directly:

```text
Developer -> S3 -> AccessDenied
```

The same developer identity successfully invoked a Lambda function whose execution role could read the object:

```text
Developer -> Lambda -> S3 -> Protected data returned
```

This proves that effective access in AWS includes not only an identity's direct permissions, but also the permissions of services and roles that identity can invoke or control.

## Test results

### Direct S3 access

The restricted developer attempted to retrieve the protected object directly with the AWS CLI.

Expected and observed result:

```text
AccessDenied: no identity-based policy allows s3:GetObject
```

### Indirect access through Lambda

The same developer invoked the Lambda function.

Observed result:

```json
{
  "statusCode": 200,
  "bucket": "brianna-iam-escalation-lab-48291",
  "object": "protected-lab-data.txt",
  "contents": "This file represents protected data that the developer should not access directly."
}
```

## Security lesson

A principal can sometimes access data indirectly through a service even when direct access is denied.

The risky permission combination explored in this lab is:

```text
lambda:CreateFunction
+
iam:PassRole
+
lambda:InvokeFunction
```

An identity with these permissions may be able to create a Lambda function, assign it a more privileged execution role, and invoke the function to perform actions outside the identity's direct permissions.

## Detection opportunities

Relevant CloudTrail events include:

- `CreateFunction`
- `UpdateFunctionCode`
- `UpdateFunctionConfiguration`
- `Invoke`
- `GetObject`
- IAM policy and role changes
- Role assumption activity

Potential detections:

- A non-administrative identity creating a Lambda function with a privileged execution role
- A developer passing a role it does not normally use
- Lambda functions created and invoked within a short time window
- Lambda functions accessing sensitive S3 buckets
- Unusual `iam:PassRole` activity

### EventBridge Detection Rule

EventBridge detected an IAM user creating a Lambda function and assigning a sensitive execution role. The event was generated by CloudTrail and forwarded to CloudWatch Logs for investigation.

The detection rule is defined in [detections/eventbridge-sensitive-lambda-role.json](detections/eventbridge-sensitive-lambda-role.json):

```json
{
  "source": ["aws.lambda"],
  "detail-type": ["AWS API Call via CloudTrail"],
  "detail": {
    "eventSource": ["lambda.amazonaws.com"],
    "eventName": ["CreateFunction20150331"],
    "userIdentity": {
      "type": ["IAMUser"]
    },
    "requestParameters": {
      "role": [
        "arn:aws:iam::ACCOUNT-ID:role/iam-escalation-lab-lambda-role"
      ]
    }
  }
}
```

## Remediation

Recommended controls include:

- Restrict `iam:PassRole` to specific role ARNs
- Use the `iam:PassedToService` condition
- Restrict which Lambda functions a developer can create or modify
- Prevent developers from passing roles with access to sensitive resources
- Review trust policies and execution-role permissions
- Monitor CloudTrail for `CreateFunction`, role passing, and sensitive S3 access
- Use permission boundaries and service control policies where appropriate
